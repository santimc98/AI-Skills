# Jira Backlog Management Skill

## Cuándo usar esta skill

Usar esta skill cuando la tarea implique crear, revisar, priorizar, documentar o actualizar trabajo en Jira.

## Objetivo

Convertir necesidades, decisiones, documentación o conversaciones en issues de Jira claros, accionables y verificables.

## Entradas necesarias

- Contexto del proyecto o producto.
- Tipo de issue: épica, historia, tarea, bug, subtarea o spike.
- Objetivo del trabajo.
- Criterios de aceptación esperados.
- Dependencias, riesgos o enlaces relacionados.

## Procedimiento paso a paso

1. Identificar el objetivo real del issue.
2. Elegir el tipo de issue adecuado.
3. Redactar título claro y orientado a acción.
4. Escribir descripción con contexto, alcance y resultado esperado.
5. Definir criterios de aceptación verificables.
6. Añadir subtareas si el trabajo es amplio.
7. Indicar dependencias, riesgos y documentación relacionada.
8. Si procede, proponer prioridad o secuencia lógica.

## Criterios de calidad

- El issue debe poder ejecutarse sin explicación adicional extensa.
- Los criterios de aceptación deben ser comprobables.
- El alcance debe estar delimitado.
- Las dependencias deben estar visibles.
- No debe mezclar demasiados objetivos en una única tarea.

## Formato de salida esperado

```markdown
## Tipo de issue

## Título

## Descripción

## Contexto

## Alcance

## Criterios de aceptación

## Subtareas

## Dependencias

## Notas
```

## Errores comunes a evitar

- Crear issues demasiado genéricos.
- No incluir criterios de aceptación.
- Mezclar tareas técnicas, funcionales y de decisión en el mismo issue.
- Crear subtareas que no aportan trazabilidad.
- No enlazar documentación relevante.
