# AI Worklog — 2026-W19

## Resumen ejecutivo

Semana centrada en la construcción de infraestructura operativa para agentes internos de Educa Edtech: organización del repositorio corporativo `ai-agents`, preparación de MCPs propios para GitLab y n8n, definición de estrategia de despliegue en Coolify/Docker, gobierno de knowledge para agentes y diseño inicial de la capa RAG con Supabase para el agente de ponencia de IA.

Se trabajó principalmente sobre cuatro líneas:

1. **MCPs internos**: validación local del MCP de n8n, construcción del MCP read-only de GitLab, pruebas con MCP Streamable HTTP, redacción de secretos, autenticación Bearer, limpieza de `node_modules` y preparación para despliegue vía Docker/Coolify.
2. **Gobernanza del repositorio corporativo de agentes**: reestructuración de `ai-agents` por categorías (`codir`, `comisiones`, `operativos`), movimiento del agente PM Ángel y del agente de ponencia, creación de manifests y documentación de taxonomía, y limpieza de carpetas no pertenecientes al repositorio corporativo.
3. **Arquitectura RAG / knowledge**: planteamiento del paso de GitLab a Supabase para el agente de ponencia, diseño de tabla vectorial propia, metadata, chunking, embeddings y estrategia de ingesta controlada desde GitLab.
4. **Preparación operativa de ingesta knowledge**: validación del GitLab Knowledge MCP desplegado, uso desde n8n mediante wrappers REST, configuración de flujo n8n para listar/leer/normalizar/chunkear knowledge y preparación de upsert en Supabase, pendiente únicamente de credenciales y prueba end-to-end.

También se analizaron automatizaciones futuras: sincronización Confluence → GitLab → RAG para knowledge de agentes y diseño conceptual de un buscador unificado B2B a partir de Zoho, Inotutor y Excel.

## Decisiones tomadas

| Fecha | Decisión | Motivo | Impacto |
|---|---|---|---|
| 2026-05-07 | Desplegar MCPs vía Docker/Coolify, no directamente desde GitLab | Los MCPs serán mantenidos principalmente por Santi y Chema; Docker simplifica despliegue y operación | GitLab queda como documentación/código fuente; Coolify/Docker como runtime real. |
| 2026-05-07 | Mantener MCP GitLab como read-only | Es suficiente para inspección de repo, knowledge, ramas y archivos; reduce riesgo operativo | No permite modificar repositorios desde el MCP. |
| 2026-05-07 | Proteger MCPs con `MCP_REQUIRE_AUTH=true` en despliegue | Evitar que endpoints MCP expuestos puedan usarse sin autorización | Las conexiones remotas deberán enviar Bearer token. |
| 2026-05-07 | No incluir `.env`, `node_modules`, `dist` ni JSON temporales en repos | Evitar exposición de secretos y ruido innecesario | Se añadieron `.gitignore` y `.dockerignore`; se limpió `node_modules` versionado. |
| 2026-05-07 | Separar agentes corporativos en `codir`, `comisiones`, `operativos` | Mejorar gobernanza, permisos, búsqueda y evolución del repositorio | El repo `ai-agents` pasa a tener taxonomía explícita. |
| 2026-05-07 | Mover agente PM Ángel a `agents/codir/` | Es un agente personalizado para directivo/perfil concreto | Queda clasificado como agente CODIR. |
| 2026-05-07 | Mover agente de ponencia a `agents/operativos/` | Es un agente especial/funcional para una ponencia y orientación a estudiantes | Queda preparado para ingesta RAG desde ruta definitiva. |
| 2026-05-07 | Crear tabla Supabase específica para el agente de ponencia | Es un agente especial y conviene aislar su knowledge | Se evita mezclar embeddings de otros agentes; se simplifica el primer MVP. |
| 2026-05-07 | Usar `text-embedding-3-small` como primera opción de embeddings | Buen equilibrio coste/calidad y dimensión estándar de 1536 | La tabla vectorial se diseñará con `vector(1536)`. |
| 2026-05-07 | No indexar `raw-research` entero sin curación | Puede contaminar respuestas con información larga, duplicada o menos validada | Se priorizan `knowledge`, `faq`, `intents`, `data` y `prompts`. |
| 2026-05-07 | Automatizar Confluence → GitLab mediante MR, no push directo a main | Mantener trazabilidad, revisión humana y rollback | n8n puede orquestar, pero la entrada a main debe pasar por revisión. |
| 2026-05-07 | Para BBDD Empresas B2B, usar base centralizada antes que consultas improvisadas por agente | El problema es de identidad/matching entre fuentes | Se recomienda PostgreSQL/Supabase + matching determinista + agente como capa de consulta. |
| 2026-05-07 | Eliminar `worklog` del repo corporativo `ai-agents` | El worklog es útil para AI-Skills personal, no para agentes corporativos | El seguimiento semanal queda en este repo personal. |
| 2026-05-07 | Eliminar `educa-bpmn-agent` del repo corporativo | No era un agente corporativo, sino una skill/procedimiento interno | El repo corporativo queda centrado en agentes, integrations, shared y MCPs. |
| 2026-05-08 | Preparar flujo GitLab MCP → n8n → Supabase con placeholders | Evitar registrar secretos y permitir dejar la arquitectura lista para ejecutar | El flujo queda pendiente solo de claves/modelo/endpoints seguros. |
| 2026-05-08 | Documentar primero GitLab corporativo antes del pipeline GitLab → Supabase | La gobernanza del repositorio y sus rutas debe estar clara antes de formalizar la ingesta RAG | Confluence priorizará GitLab e infraestructura MCP; pipeline RAG se documentará tras prueba end-to-end. |

## Riesgos / bloqueos

- Docker Desktop está bloqueado por permisos/política corporativa; sin Docker local no se pueden construir imágenes desde el portátil hasta que Soporte/IT lo resuelva o se use build en servidor.
- Los MCPs no deben exponerse sin autenticación Bearer ni sin HTTPS/proxy seguro.
- Las variables `N8N_API_KEY`, `GITLAB_TOKEN` y `MCP_API_KEY` deben vivir solo en Coolify/entorno seguro, nunca en Git.
- Las claves de OpenAI y Supabase no deben registrarse en worklogs, Jira, Confluence ni capturas compartidas si no es necesario.
- Si `.env` llegó a subirse a un remoto en algún momento, hay que rotar tokens aunque luego se haya eliminado.
- El repo corporativo `ai-agents` debe mantenerse libre de `node_modules`, `dist`, `.env`, `.claude/worktrees` y worklogs personales.
- La ingesta RAG desde GitLab debe usar rutas definitivas; si se indexa antes de cerrar estructura, habrá que reindexar.
- Para agentes con knowledge procedente de Confluence, el push directo a `main` puede introducir errores sin revisión; mejor rama + MR.
- En el buscador B2B, un matching fuzzy demasiado bajo no debe aceptar automáticamente equivalencias de empresa; los casos dudosos requieren revisión humana.
- No conviene documentar formalmente el pipeline GitLab → Supabase en Confluence hasta validarlo end-to-end.
