## Decisiones tomadas

| Fecha | Decisión | Motivo | Impacto |
|---|---|---|---|
| 2026-04-27 | Usar GitHub como fuente neutral de skills | Evitar dependencia exclusiva de Claude o ChatGPT | Las skills son reutilizables entre asistentes |
| 2026-04-27 | No conectar SharePoint todavía | Los permisos solicitados son demasiado amplios | La documentación de SharePoint se compartirá manualmente por ahora |
| 2026-04-27 | Crear skill específica de Jira | El proyecto AA tiene reglas propias de Epics, labels, estados y documentación | Mejora la consistencia al trabajar con Jira |
| 2026-04-27 | Crear worklog común | Claude y ChatGPT necesitan una memoria operativa compartida | Facilita resumen semanal, continuidad y creación posterior de Jira/Confluence |
| 2026-04-27 | Tratar el sistema IMO como MVP metodológico | Hay buena base, pero requiere validación de ponderaciones y separación conceptual | Evita usar el scoring como criterio definitivo antes de revisión del owner responsable |
| 2026-04-27 | Separar IMO, IRPI e IPA | Madurez, riesgo y oportunidad de automatización responden a decisiones distintas | Mejora la lectura ejecutiva y priorización de automatizaciones/agentes |
| 2026-04-27 | No actualizar aún la página oficial de scoring | Está pendiente de visto bueno del owner responsable | Se dejó comentario/revisión, pero no se reemplazó la documentación oficial |
| 2026-04-27 | Para Confluence importante, usar estrategia A/B/C/D | El reemplazo completo puede ser más riesgoso para un LLM con herramientas | Reduce riesgo de sobrescritura y pérdida de formato/macro/contenido |
| 2026-04-27 | Al editar Excel, preservar estilo del workbook original | Las nuevas hojas deben parecer parte del mismo archivo | Se actualizó la skill Excel para exigir paleta/formatos coherentes |
| 2026-04-27 | Minimizar datos personales en el repo público | Evitar exposición innecesaria de nombres/emails corporativos | Se prioriza el uso de roles funcionales: owner responsable, responsable Tecnología, responsable IT |
| 2026-04-29 | Usar autoría interna en Confluence | La autoría visible debe reflejar responsabilidad documental interna, no la herramienta usada | En históricos y metadatos se usará `Santiago Martín` como autor documental cuando aplique |
| 2026-04-29 | Documentar automatizaciones respetando el trabajo previo del compañero | Evitar sobrescribir criterio funcional ya rellenado por quien está documentando | Solo se completan huecos técnicos observables en n8n; si algo choca, se mantiene lo escrito por el compañero |
| 2026-04-29 | Mantener como pendientes los campos no confirmados | No inventar owners, IDs AUT, políticas de error, credenciales ni decisiones no cerradas | Las páginas quedan útiles como borrador técnico sin afirmar información no validada |

## Riesgos / bloqueos

- SharePoint no debe conectarse sin validación de IT por el alcance amplio de permisos.
- El repositorio `AI-Skills` es público; evitar información confidencial, emails corporativos y nombres de compañeros cuando baste con roles.
- Las acciones reales en Jira/Confluence deben requerir confirmación explícita antes de ejecutarse.
- La página oficial del scoring IMO no debe actualizarse hasta que el owner responsable valide la propuesta metodológica.
- El Excel v2 generado mejora la metodología, pero Santi detectó que las nuevas hojas deben respetar mejor estilo/paleta del archivo original; la skill Excel ya fue corregida para futuras iteraciones.
- Las fichas de automatización n8n contienen campos técnicos pendientes: IDs AUT, owner técnico, políticas reales de reintentos, notificaciones de fallo, community nodes y credenciales de referencia.
- Las capturas de n8n permiten documentar flujo visible, pero no sustituyen revisión técnica del workflow ni revisión de executions/logs.
